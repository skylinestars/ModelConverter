# 编码规范

## 语言与标准
- **C++17**，优先使用标准库算法。
- **禁用 C 风格宏、裸 `new/delete`**，使用 `std::unique_ptr` / `std::shared_ptr`。
- **RAII 优先**。

## 内存与跨接口约定
- 跨 DLL 接口、可序列化结构中**避免暴露 `std::shared_ptr`**。
- 内部所有权用 `std::unique_ptr`；跨对象引用用**裸指针、Handle 或 ID**。
- 这正是项目使用 `ObjectID`（`uint64_t`）而非指针做跨组件引用的原因之一。

## 类文件编码规范
### 保持单一职责
新增函数必须与当前类职责相关，不允许加入无关功能。
如果功能可以独立复用，应新建工具类或独立模块。

### 控制函数长度
单个函数建议不超过 200 行。
超过 200 行时，应拆分为多个私有辅助函数。

### 保持接口简洁
Public 函数只负责业务流程，不要堆积实现细节。
复杂逻辑应放入 Private 函数。

### 一个函数只做一件事
不要在一个函数中同时完成解析、校验、转换、存储等多个步骤。
每个函数应只有一个明确职责。

### 控制缩进层级
缩进不得超过 3 层。
优先使用提前 return（Early Return）减少嵌套。

### 不新增无意义成员变量
优先使用局部变量。
仅当状态需要跨多个成员函数共享时，才新增成员变量.

### 不写重复代码
已有相同逻辑时，应复用已有函数，不允许复制粘贴代码.

### 必要时添加注释
仅解释"为什么这样实现"，不要解释代码本身在做什么.

### 修改后保持文件整洁
新增代码应放在逻辑相近的位置。
不要随意打乱已有函数顺序.

### 不允许继续扩大 God Class
如果新增功能导致类职责明显增加，应建议拆分新类，而不是继续向当前类添加函数。

## 命名规范

函数名必须使用动词，如：
ParseXXX()
LoadXXX()
BuildXXX()
ValidateXXX()
不要使用 Process()、Handle()、DoWork() 等含义模糊的名字。

| 类型 | 规范 | 示例 |
|---|---|---|
| 类/类型 | 帕斯卡 | `Importer`、`aiScene` |
| 函数 | 帕斯卡 | `ReadFile`、`GetExtension` |
| 变量 | 驼峰 | `scene`、`importStep` |
| 常量 | `k` 驼峰或大写蛇形 | `kMaxVertices` |
| 成员变量 | `m_` 前缀 | `mScene` |
| 静态变量 | `s_` 前缀 | — |

## 格式与注释
- 编码：**UTF-8**；缩进：**4 空格**；换行符：**LF**。
- 注释**使用中文**，英文注释仅用于特殊逻辑或性能优化说明。
