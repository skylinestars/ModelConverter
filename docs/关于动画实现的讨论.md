这两个设计实际上代表了两种完全不同的动画架构思路：

## 方案1：技术架构文档（Typed Animation）

```cpp
NodeAnimation
{
    translation;
    rotation;
    scale;
}

MorphAnimation
{
    weights;
}
```

特点：

```text
强类型
易读
易调试
符合引擎思维
符合游戏引擎常见实现
```

例如：

```cpp
channel.translation[0].value
channel.rotation[0].value
```

一眼就知道是什么数据。

---

## 方案2：DeepSeek（GLTF风格）

```cpp
AnimationChannel
{
    targetPath;
    sampler;
}
```

本质上是在模仿 GLTF：

```json
{
  "channels": [
    {
      "target": {
        "node": 1,
        "path": "translation"
      },
      "sampler": 0
    }
  ]
}
```

对应：

```cpp
AnimationChannel
{
    nodeId;
    targetPath = Translation;
    sampler;
}
```

特点：

```text
更通用
更接近文件格式
更接近USD
更接近GLTF
但不适合作为Core层
```

---

# 先说结论

对于你的项目：

```text
三维格式转换器
Core Scene
↓
FBX
GLTF
USD
...
```

我推荐：

> **Core层采用技术架构文档方案（Typed Animation）**
>
> Importer/Exporter内部采用DeepSeek方案（Sampler/Channel）

不要反过来。

# 为什么不建议直接采用 DeepSeek 的 Core

假设导入一个 FBX：

```cpp
AnimationChannel
{
    nodeId = 123;
    targetPath = Translation;

    sampler.values =
    {
        0,0,0,
        1,0,0,
        2,0,0
    };
}
```

你要取第2帧：

```cpp
size_t offset = keyIndex * 3;
```

取：

```cpp
values[offset + 0]
values[offset + 1]
values[offset + 2]
```

才能得到：

```cpp
Vec3
```

---

如果是 Rotation：

```cpp
targetPath = Rotation
```

又变成：

```cpp
x,y,z,w
```

stride=4

---

如果是 Morph：

```cpp
targetPath = Weights
```

又变成：

```cpp
weight0
weight1
weight2
...
```

stride不固定

---

于是整个 Core 会充满：

```cpp
switch(targetPath)
{
case Translation:
case Rotation:
case Scale:
case Weight:
}
```

---

这就是典型的：

```text
文件格式友好
程序开发不友好
```

struct NodeAnimation

enum class CurveValueType

# 关于 Interpolation

这个是我认为技术架构文档必须补的

你当前：

```cpp
KeyFrameVec3
{
    time;
    value;
}
```

缺少：

```cpp
Interpolation
```

而 GLTF：

```json
LINEAR
STEP
CUBICSPLINE
```

FBX：

```text
Constant
Linear
Cubic
Bezier
```

USD：

```text
Held
Linear
```

都有插值信息。

---

我建议增加：

```cpp
enum class AnimationInterpolation
{
    Step,
    Linear,
    CubicSpline
};
```

---

然后：

```cpp
struct NodeAnimation
{
    AnimationInterpolation translationInterpolation;
    AnimationInterpolation rotationInterpolation;
    AnimationInterpolation scaleInterpolation;

    ...
};
```

或者：

```cpp
struct AnimationTrack<T>
{
    AnimationInterpolation interpolation;

    std::vector<KeyFrame<T>> keys;
};
```

更优雅。

# 最终建议

如果是你的转换器项目：

### 不推荐

DeepSeek方案直接作为 Core

```cpp
AnimationChannel
AnimationSampler
packed values
```

原因：

```text
过于底层
接近GLTF
不利于后续Pass开发
```

---

### 推荐

在你当前方案基础上升级：

```cpp
AnimationTrack<T>

NodeAnimation

MorphAnimation

AnimationClip
```

例如：

```cpp
template<typename T>
struct KeyFrame
{
    double time;
    T value;
};

template<typename T>
struct AnimationTrack
{
    AnimationInterpolation interpolation;

    std::vector<KeyFrame<T>> keys;
};
```

然后：

```cpp
struct NodeAnimation
{
    ObjectID nodeId;

    AnimationTrack<Vec3> translation;
    AnimationTrack<Quaternion> rotation;
    AnimationTrack<Vec3> scale;
};
```

这是我认为最适合作为你这个 **Scene Core 层** 的设计。

而 DeepSeek 那套：

```cpp
AnimationSampler
AnimationChannel
```

保留在：

```text
Plugin_Gltf
Plugin_Fbx
Plugin_Usd
```

内部实现层即可，不要污染 Core 数据结构。这样后续做 AnimationOptimizePass、AnimationCompressionPass、AnimationResamplePass、FBX→GLTF 动画转换都会舒服很多。
